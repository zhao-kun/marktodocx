# Design Document: markdocx Chrome Extension

## 1. Background

### 1.1 Current State

markdocx is a Node.js CLI tool (`md-to-docx.mjs`) that converts Markdown files to Word (.docx) documents. It supports headings, lists, tables, code blocks, local images, and Mermaid diagrams.

The current conversion pipeline:

```
Markdown file (fs.readFile)
    → markdown-it (parse & render to HTML)
    → Mermaid CLI (mmdc + Puppeteer + Chrome subprocess → PNG)
    → sharp (trim PNG whitespace)
    → jsdom (inline local images as base64, normalize tables)
    → html-to-docx (HTML → DOCX buffer)
    → fs.writeFile (.docx)
```

### 1.2 Motivation

The CLI tool requires Node.js 22+, npm, and a local Chrome binary installed via Puppeteer. This setup is non-trivial for non-developer users. A Chrome extension would:

- Eliminate the Node.js/npm/Puppeteer setup entirely
- Allow users to convert Markdown to DOCX from the browser with one click
- Leverage Chrome itself for Mermaid rendering instead of spawning a subprocess
- Be distributable via Chrome Web Store

### 1.3 Scope

This document covers the feasibility analysis, dependency audit, architecture design, and implementation plan for porting markdocx from a Node.js CLI tool to a Chrome extension.

## 2. Feasibility Analysis

### 2.1 Dependency Audit

| Dependency | Version | Browser Compatible | Assessment |
|---|---|---|---|
| `markdown-it` | 14.1.1 | Yes | Pure JavaScript. Works in browser without modification. |
| `mermaid` | 11.14.0 | Yes | Designed for browser. Can render SVG directly into DOM elements. **Simpler** than current CLI approach. |
| `@mermaid-js/mermaid-cli` | 11.12.0 | No | Wraps Puppeteer to launch Chrome. Unnecessary — we are already inside Chrome. **Drop entirely.** |
| `sharp` | 0.34.5 | No | Native C++ addon (libvips). Used only for `trim()` on Mermaid PNGs. **Replace with Canvas API.** |
| `jsdom` | 29.0.2 | No (unnecessary) | DOM manipulation library for Node.js. In a browser extension, native DOM is available. **Drop entirely.** |
| `html-to-docx` | 1.8.0 | Partial | Bundles `node-fetch` and uses `Buffer` (64 occurrences). Internally uses JSZip + xmlbuilder2 which are browser-compatible. **Requires polyfill or replacement.** |
| `dompurify` | 3.3.3 | Yes | Pure JavaScript. Works in browser. |

### 2.2 Critical Risk: html-to-docx

`html-to-docx` is the most complex dependency to evaluate. Findings:

- **Buffer usage**: 64 occurrences in the bundled ESM output. Chrome extensions do not provide Node.js `Buffer`. However, `Buffer` can be polyfilled with `Uint8Array`-based shims (e.g., via bundler config).
- **node-fetch**: Bundled inline. Used for remote image fetching. In a Chrome extension, this can be replaced with native `fetch()`.
- **JSZip**: Browser-compatible. This is the core DOCX packaging engine.
- **xmlbuilder2**: Browser-compatible. Used to construct XML inside the DOCX.

**Assessment**: `html-to-docx` *can* work in a browser extension with a bundler that polyfills `Buffer` and replaces `node-fetch` with native `fetch`. If polyfilling proves fragile, the fallback is to switch to the `docx` npm package (fully browser-native) at the cost of rewriting the HTML→DOCX step.

### 2.3 Feasibility Verdict

**Feasible.** All blocking dependencies have clear browser alternatives. The hardest part is `html-to-docx` browser compatibility, which has two viable paths (polyfill or replace).

## 3. Possible Solutions

### 3.1 Option A: Polyfill html-to-docx (Low Effort, Some Risk)

Keep `html-to-docx` as the DOCX generator. Use a bundler (Webpack/Vite) to polyfill `Buffer` and alias `node-fetch` to native `fetch`.

**Pros:**
- Minimal changes to existing conversion logic
- Code block rendering, table normalization, and styling code can be reused as-is
- Fast to implement

**Cons:**
- Fragile — `html-to-docx` was not designed for browser use and may have hidden Node.js assumptions
- Future `html-to-docx` updates could break browser compatibility
- Polyfill adds bundle size (~50KB for Buffer shim)

**Risk**: Medium. May encounter edge cases where `Buffer` polyfill behaves differently.

### 3.2 Option B: Replace with docx Library (High Effort, Low Runtime Risk)

Replace `html-to-docx` with the [`docx`](https://www.npmjs.com/package/docx) npm package, which is fully browser-native and builds DOCX documents programmatically.

**Pros:**
- `docx` is designed for browser and Node.js — no polyfills needed
- Full control over DOCX output (styles, spacing, page layout)
- More reliable long-term
- Smaller bundle

**Cons:**
- **Major rewrite, not a swap.** The current pipeline is entirely HTML-driven: code blocks are rendered as styled `<table>` elements (lines 102-125), table widths and blockquote styles are normalized via DOM manipulation (lines 295-343), and the final DOCX is generated by passing a complete HTML document to `html-to-docx` (lines 398-408). Switching to `docx` means rebuilding all of this as programmatic `Paragraph`/`Table`/`ImageRun` objects.
- The "60% reuse" estimate is optimistic — the reusable parts are small utility functions, while the heavy rendering logic is HTML-coupled and must be rewritten.
- Introduces two major changes simultaneously (new platform + new renderer), making it harder to diagnose issues.

**Risk**: Low runtime risk (no polyfills needed), but **high output-fidelity and rewrite-scope risk**. The effort is closer to "high" than "medium".

### 3.3 Option C: Hybrid Extension with Offscreen Document (Medium Effort, Low Risk)

Use Chrome's [Offscreen Documents API](https://developer.chrome.com/docs/extensions/reference/api/offscreen) to run the conversion in a hidden document context. This provides a full DOM environment for Mermaid rendering and HTML manipulation.

**Pros:**
- Clean separation: popup handles UI, offscreen document handles conversion
- Full DOM available for Mermaid rendering and HTML processing
- Can combine with either Option A or B for DOCX generation

**Cons:**
- Offscreen Documents API requires Manifest V3
- Only one offscreen document can exist at a time
- Slightly more complex extension architecture

**Risk**: Low. This is the recommended Chrome extension pattern for DOM-heavy background work.

### 3.4 Recommendation

**Option C (Offscreen Document) + Option A (polyfill html-to-docx) for v1**, with Option B as a later refactor.

Rationale:
- Offscreen Document is the right extension-side execution model for Mermaid and DOM work.
- **Do not combine a new platform with a new renderer in v1.** The current converter is entirely HTML-centric: code blocks render as styled `<table>` elements, table normalization and blockquote styling are done via DOM manipulation, and `html-to-docx` consumes a complete HTML document. Switching to the `docx` library means rewriting all of this as programmatic `Paragraph`/`Table`/`ImageRun` objects — that is a full renderer rewrite, not a "medium effort" swap.
- Shipping Option C + A first keeps one major variable (the platform) while preserving the proven rendering pipeline. This de-risks the port.
- Keep Option B as a follow-up refactor only if html-to-docx proves unstable in MV3 or if deeper DOCX styling control is needed.

**Required before committing to Option A:** Time-box a small spike (1-2 days) to prove `html-to-docx` works inside an MV3 offscreen document with `Buffer` polyfill under extension CSP. If the spike fails, fall back to Option B — but do so with eyes open about the rewrite scope.

## 4. Proposed Architecture

### 4.1 Extension Structure

```
markdocx-extension/
├── manifest.json              # Manifest V3 config
├── background.js              # Service worker: offscreen lifecycle, message routing, error handling
├── popup/
│   ├── popup.html             # Upload UI (directory picker + convert button)
│   ├── popup.css              # Styles
│   └── popup.js               # UI logic, directory/image resolution, triggers conversion
├── offscreen/
│   ├── offscreen.html         # Hidden DOM for Mermaid rendering
│   └── offscreen.js           # Conversion pipeline (receives markdown + image data)
├── lib/
│   ├── md-renderer.js         # markdown-it setup + custom renderers (from md-to-docx.mjs)
│   ├── mermaid-renderer.js    # Mermaid SVG → PNG via Canvas
│   ├── docx-generator.js      # HTML → DOCX via html-to-docx (v1, polyfilled)
│   ├── image-resolver.js      # Resolve relative image paths from directory handle
│   └── table-normalizer.js    # Table width normalization (from md-to-docx.mjs)
├── icons/                     # Extension icons
└── vite.config.js             # Bundler config (Buffer polyfill, node-fetch alias)
```

### 4.2 Conversion Pipeline

```
User selects folder containing .md file and images (directory picker)
    ↓
popup.js reads .md file + resolves relative image paths from directory
    ↓
popup.js inlines local images as base64 data URIs
    ↓
popup.js sends { markdown, imageMap } to service worker
    ↓  (chrome.runtime.sendMessage)
service worker creates offscreen document (if not already open)
    ↓
service worker forwards message to offscreen document
    ↓
offscreen.js receives Markdown text + image data
    ↓
markdown-it parses Markdown, extracts Mermaid blocks
    ↓
mermaid.render() → SVG string (in offscreen DOM)
    ↓
SVG → Canvas → PNG blob (trim whitespace via pixel scanning)
    ↓
markdown-it renders full HTML with Mermaid PNGs and images injected
    ↓
DOM manipulation: normalize tables, style blockquotes
    ↓
html-to-docx: convert HTML document to DOCX buffer (v1, polyfilled)
    ↓
DOCX Blob sent back via service worker to popup.js
    ↓
service worker closes offscreen document
    ↓
chrome.downloads.download({ url: blobUrl, filename: '*.docx' })
```

### 4.3 Mermaid Rendering (Browser)

Current CLI approach:
```
Mermaid code → write .mmd file → mmdc subprocess → Puppeteer → Chrome → PNG file → sharp trim → read file
```

Proposed browser approach:
```
Mermaid code → mermaid.render(id, code) → SVG string
    → create Image from SVG blob URL
    → draw onto Canvas
    → scan pixels to find bounding box (replaces sharp.trim())
    → canvas.toBlob('image/png')
    → data URL
```

This is fundamentally simpler. No subprocess, no temp files, no native addon.

### 4.4 Message Flow

```
┌──────────┐    chrome.runtime    ┌────────────────┐    chrome.runtime    ┌───────────────────┐
│  Popup   │ ── sendMessage ────→ │ Service Worker  │ ── sendMessage ───→ │ Offscreen Document │
│  (UI)    │ ←─ sendResponse ──── │ (Background)    │ ←─ sendResponse ─── │ (Conversion)       │
└──────────┘                      └────────────────┘                      └───────────────────┘
     │                                   │                                        │
     │ Directory picker                  │ createDocument()                       │ mermaid.render()
     │ Image resolution                  │ closeDocument()                        │ Canvas API
     │ chrome.downloads                  │ Error routing                          │ html-to-docx (v1)
     ▼                                   ▼                                        ▼
  User's folder                    Lifecycle mgmt                           DOCX Blob
```

### 4.5 Manifest V3 Permissions

```json
{
  "manifest_version": 3,
  "name": "markdocx",
  "version": "1.0.0",
  "description": "Convert Markdown to Word (.docx) with Mermaid diagram support",
  "permissions": [
    "offscreen",
    "downloads"
  ],
  "background": {
    "service_worker": "background.js",
    "type": "module"
  },
  "action": {
    "default_popup": "popup/popup.html",
    "default_icon": "icons/icon-128.png"
  }
}
```

No special host permissions or content script injection needed. The extension operates entirely on user-provided files.

### 4.6 Offscreen Document Lifecycle

The message flow in §4.4 omits who creates the offscreen document and manages its lifecycle. This must be handled by a **background service worker**:

```
┌──────────┐   chrome.runtime    ┌────────────────┐   chrome.offscreen    ┌───────────────────┐
│  Popup   │ ── sendMessage ──→  │ Service Worker  │ ── createDocument ──→ │ Offscreen Document │
│  (UI)    │ ←─ sendResponse ──  │ (Background)    │ ←─ sendMessage ─────  │ (Conversion)       │
└──────────┘                     └────────────────┘                       └───────────────────┘
```

The service worker is responsible for:

1. **Creating the offscreen document** on demand via `chrome.offscreen.createDocument()` when a conversion is requested.
2. **Routing messages** between the popup and the offscreen document.
3. **Closing the offscreen document** after conversion completes (or on timeout/failure) via `chrome.offscreen.closeDocument()`. Only one offscreen document can exist at a time per extension.
4. **Error routing** — if the offscreen document crashes or the conversion throws, the service worker catches the failure and routes an error response back to the popup for user display.

The manifest must include a `background` entry:

```json
{
  "background": {
    "service_worker": "background.js",
    "type": "module"
  }
}
```

### 5.1 Critical: Local Image Support Requires Directory Access

The current CLI tool resolves relative image paths from the Markdown file's parent directory:

```javascript
// md-to-docx.mjs:267
const absolutePath = path.resolve(baseDir, decodedSrc);
const fileBuffer = await fs.readFile(absolutePath);
```

A browser extension **cannot** replicate this with a simple `FileReader` on the dropped `.md` file. `FileReader` only works on `File` objects the user has explicitly selected — it cannot resolve `./images/diagram.png` relative to the Markdown file's location on disk.

**Solution:** The popup must use `<input type="file" webkitdirectory>` or the [File System Access API](https://developer.chrome.com/docs/capabilities/web#file_system_access) (`window.showDirectoryPicker()`) to let the user select the **folder** containing the Markdown file and its images. The extension then:

1. Receives a `FileSystemDirectoryHandle` or a flat list of `File` objects with `webkitRelativePath`.
2. Resolves relative image `src` paths against the directory tree.
3. Reads matched image files and converts them to base64 data URIs.

This affects the popup UI (must prompt for folder, not just a single file), the message protocol (must pass image data alongside Markdown text to the offscreen document), and the conversion pipeline.

**If this is not designed in, local images will silently break.** This is the most significant architecture gap in the current plan.

### 8.1 Spike Results

**Status: PASSED**

The spike is implemented in `markdocx-extension/`. All checklist items in §9 verified.

**What worked:**

- `html-to-docx` runs inside an MV3 offscreen document with `vite-plugin-node-polyfills` providing `Buffer`, `stream`, `util`, `path`, `url`, `crypto`, and `process` polyfills.
- Generated DOCX (42KB) contains valid content: heading, paragraph with inline formatting (bold, italic, code), unordered list, ordered list, 3-column table, and styled code block.
- Full message flow works: popup → service worker → offscreen → service worker → popup.
- Offscreen document lifecycle (create on demand, close after conversion) works correctly.
- No extension CSP violations.

**Issues encountered and resolved:**

1. **`URL.createObjectURL` not available in service workers.** Service workers lack DOM APIs. Fixed by moving blob creation and download trigger to the popup instead.
2. **`new Uint8Array(polyfillBuffer)` produces empty output.** The polyfilled `Buffer` is a `Uint8Array` subclass, but `new Uint8Array(typedArray)` copies the length, not the bytes. Fixed by using `new Uint8Array(data.buffer, data.byteOffset, data.byteLength)` to extract from the underlying `ArrayBuffer`.
3. **`vm` module polyfill contains `eval()`.** The `vm-browserify` polyfill uses direct `eval()`, which is blocked by extension CSP. Excluded `vm` from polyfills — it's only used by `asn1.js` in the crypto polyfill chain, not in the DOCX generation hot path.
4. **Vite HTML output paths don't match extension manifest expectations.** Vite preserves the source directory structure for HTML files. Fixed by placing HTML files in `public/` (static, copied as-is) and only bundling JS entry points through Vite.

**Build output:**

| File | Size | Notes |
|---|---|---|
| `popup/popup.js` | 0.8 KB | Popup UI logic |
| `background.js` | 0.8 KB | Service worker |
| `offscreen/offscreen.js` | 2.1 MB (577 KB gzip) | html-to-docx + all polyfills |

The offscreen bundle is large due to polyfills. Acceptable for v1 — can be optimized later by tree-shaking unused polyfill paths.

## 9. Phase 5: Code Block Syntax Highlighting

### 9.1 Current State

Code blocks are rendered as plain monochrome text inside a styled `<table>` cell (`md-renderer.js:renderCodeBlockHtml`). All code is the same color (#111827 body text), with only a gray background (#F0F0F0) and a language badge distinguishing code from prose. There is no syntax highlighting — keywords, strings, comments, and other tokens are visually indistinguishable.

### 9.2 Goal

Add syntax highlighting for **Python, Go, Java, Bash, JavaScript, and TypeScript** code blocks. Tokens (keywords, strings, comments, numbers, etc.) should render with distinct inline colors that survive the HTML → DOCX conversion via `html-to-docx`.

### 9.3 Approach: highlight.js with Inline Styles

**Library choice: [highlight.js](https://highlightjs.org/)**

- Pure JavaScript, browser-compatible, no dependencies
- Supports all six target languages out of the box
- Produces `<span class="hljs-keyword">...</span>` markup that can be mapped to inline styles
- Tree-shakeable: can import only the needed language grammars to minimize bundle size
- Well-maintained, widely used (GitHub, Stack Overflow)

**Why not alternatives:**
- `Prism.js` — similar capability, but highlight.js has better markdown-it integration via `markdown-it-highlightjs` or direct API usage
- `shiki` — uses TextMate grammars for higher fidelity, but depends on WASM (oniguruma) which is blocked by extension CSP (`wasm-eval` not allowed). Not viable in MV3.

### 9.4 Integration Point

The highlighting hooks into the existing `renderCodeBlockHtml()` function in `md-renderer.js`. Currently this function calls `preserveCodeWhitespace()` on each raw line. With highlighting:

```text
Before (current):
  raw code string → split lines → escapeHtml → preserveCodeWhitespace → monochrome HTML

After (with highlighting):
  raw code string → hljs.highlight(code, {language})
    → highlighted HTML with <span> classes
    → convert CSS classes to inline styles (required for html-to-docx)
    → line-safe split (see §9.4.1)
    → whitespace preservation on text nodes only (see §9.4.2)
    → wrap in .code-block-line divs
```

**Critical constraint:** `html-to-docx` does not process `<style>` blocks or CSS classes for inline element colors. All colors must be applied as `style="color: #xxx"` attributes directly on `<span>` elements. This means we cannot use highlight.js's CSS themes directly — we must map `hljs-*` classes to inline style attributes after highlighting.

#### 9.4.1 Line Splitting with Multi-Line Tokens

highlight.js tokens can span multiple lines. For example, a multi-line string or block comment produces a single `<span>` whose text content contains `\n` characters:

```html
<span class="hljs-comment">/* line 1
line 2
line 3 */</span>
```

Naively splitting the highlighted HTML on `\n` would produce unbalanced markup (opening `<span>` on one line, closing `</span>` on another). The splitter must track open `<span>` tags across line breaks:

1. Walk the highlighted HTML character by character, tracking a stack of currently open `<span>` tags (with their attributes).
2. On encountering `\n`, close all open spans, emit the completed line, start a new line, and re-open the spans from the stack.
3. On `<span ...>`, push onto the stack. On `</span>`, pop.
4. After the final character, emit the last line.

This produces per-line HTML fragments where every line has balanced `<span>` tags, even when the original token crossed lines.

#### 9.4.2 Whitespace Preservation After Highlighting

The current `preserveCodeWhitespace()` in `md-renderer.js:13` calls `escapeHtml()` before replacing spaces and tabs with `&nbsp;`. If applied to already-highlighted HTML, it would escape `<span>` tags into literal `&lt;span&gt;` text, destroying all highlighting.

The highlighted output must use a **tag-aware** whitespace pass instead:

- Walk each line's HTML, distinguishing tag markup (`<span ...>`, `</span>`) from text content.
- Only replace spaces → `&nbsp;` and tabs → `&nbsp;&nbsp;&nbsp;&nbsp;` inside text content segments.
- Leave tag markup untouched.

This can be implemented with a simple state machine or regex that matches either HTML tags or inter-tag text, applying substitutions only to the text matches.

For **unhighlighted** code blocks (unsupported language or no language specified), the existing `escapeHtml` + `preserveCodeWhitespace` path is still used — no change needed.

### 9.5 Color Theme

Use a GitHub-like light theme that reads well on the #F0F0F0 code block background:

**highlight.js sub-scope class format:** highlight.js v11 emits sub-scoped tokens as a single `<span>` with multiple CSS classes. The first scope segment gets the `hljs-` prefix; subsequent segments get a trailing underscore instead. Examples:

- Scope `title.function` → `<span class="hljs-title function_">`
- Scope `variable.language` → `<span class="hljs-variable language_">`
- Scope `title.class.inherited` → `<span class="hljs-title class_ inherited__">`

The color map uses the **exact class attribute value** as the key for compound entries. The resolver looks up the full `class` value first (compound match), then falls back to the first class only (primary match). If neither matches, the span is unwrapped.

```
└── syntax-highlighter.js   # NEW: highlight.js setup + class-to-inline-style conversion
```

**`syntax-highlighter.js` responsibilities:**

1. Import highlight.js core and six language grammars (python, go, java, bash, javascript, typescript)
2. Register the languages (including `js` → javascript and `ts` → typescript aliases)
3. Export `highlightCode(code, language)` which:
   - Returns `null` if the language is not supported (caller falls back to monochrome path)
   - Calls `hljs.highlight(code, { language, ignoreIllegals: true })` for supported languages. The `ignoreIllegals` flag is required because code fences commonly contain partial snippets, pseudo-code, or fragments that would otherwise cause highlight.js to throw an `IllegalError`. If highlighting still fails despite this flag (e.g. grammar bug), catch the error and return `null` to fall back to monochrome.
   - Converts `class` attributes to `style="color: #xxx"` via regex. highlight.js v11 emits single-class spans (`class="hljs-keyword"`) and multi-class sub-scoped spans (`class="hljs-title function_"`, `class="hljs-variable language_"`) — see §9.5 for the exact class format. The resolver extracts the full `class` attribute value, looks it up in the color map as-is (compound match), then falls back to the first class only (primary match). If neither matches, the span is unwrapped (removed) so the text inherits the default code block color.
   - Splits lines safely using the span-tracking algorithm from §9.4.1
   - Applies tag-aware whitespace preservation from §9.4.2
   - Returns an array of per-line HTML strings, each with balanced tags and preserved whitespace

**`md-renderer.js` changes:**

- `renderCodeBlockHtml()` calls `highlightCode(content, language)`
- If `highlightCode` returns an array (supported language): use the pre-processed lines directly, skip `escapeHtml`/`preserveCodeWhitespace`
- If `highlightCode` returns `null` (unsupported language): use the existing monochrome path unchanged

### 9.8 Scope and Non-Goals

**In scope:**

- Syntax highlighting for Python, Go, Java, Bash, JavaScript, TypeScript fenced code blocks
- Common aliases supported: `js` → javascript, `ts` → typescript
- Colors survive into DOCX output via inline styles
- Unsupported languages fall back to current monochrome rendering (no error)

**Not in scope:**

- Auto-detection of language when no fence info string is provided
- Line numbers in code blocks
- Custom theme configuration by users
- Additional languages beyond the six listed (can be added later by registering more grammars)
